# Internationalization (i18n)

> All user-facing strings go through i18next. `en` and `zh` locale files must
> stay structurally identical. This is a real, must-follow invariant in this
> repo — the renderer ships bilingual.

---

## Setup

`src/renderer/i18n/config.ts` initializes i18next with:

- `initReactI18next` (react-i18next binding) and `LanguageDetector`
  (`i18next-browser-languagedetector`).
- `resources` = `{ en: { translation: enTranslations }, zh: { translation:
zhTranslations } }`, imported from `locales/en.json` and `locales/zh.json`.
- `fallbackLng: 'en'`, `supportedLngs: ['en', 'zh']`.
- Language detection order `['localStorage', 'navigator']`, cached in
  `localStorage` under `i18nextLng`.
- `pluralSeparator: '_'` (so plurals use `key` + `key_plural`).

The module is initialized once, imported for its side effect at the top of
`src/renderer/main.tsx` (`import './i18n/config'`). It also `export default i18n`
for the singleton (see below).

## Usage in components

```tsx
const { t } = useTranslation();
return <span>{t('sidebar.newTask')}</span>;
```

- Interpolation: `t('api.testSuccess', { ms })` → the JSON value uses
  `{{ms}}` (see `locales/en.json`, e.g. `"testSuccess": "Connection successful
({{ms}}ms)"`).
- Plurals: define `key` and `key_plural` with a `{{count}}` variable and call
  `t('key', { count })`.

~49 renderer components use `useTranslation`. New user-facing text must not be a
hard-coded string literal.

---

## Non-component code uses the singleton

Code that isn't a React component imports the singleton instead of the hook:

```ts
import i18n from '../i18n/config';
i18n.t('chat.startFailed');
```

Real call sites: `useIPC.ts:14` (error/notice messages),
`utils/i18n-format.ts:1`, `components/ErrorBoundary.tsx:4`. The store's
`GlobalNotice` carries a `messageKey` + `messageValues` so the message can be
(re)translated by key rather than storing a frozen English string.

## THE invariant: en/zh structural parity

`locales/en.json` and `locales/zh.json` must have the **identical nested key
structure** (same top-level namespaces, same keys under each). They currently
share 28 top-level namespaces: `api, chat, common, compaction, context,
credentials, errorBoundary, general, language, logs, mcp, memory, messageCard,
nav, permission, remote, sandbox, schedule, settings, sidebar, skills, subagent,
sudo, time, toolUse, welcome, window`.

**Rule:** every new user-facing string is added to BOTH `en.json` AND `zh.json`
at the same key path, in the same commit.

**Anti-patterns:**

- Hard-coded UI strings in JSX/TSX instead of `t('...')`.
- Adding a key to only one locale (breaks parity; falls back to `en` silently and
  ships an untranslated string in the other language).
- Diverging the nested shape between the two files.

## Quick reference

| Need                         | Do this                                               |
| ---------------------------- | ----------------------------------------------------- |
| String in a component        | `const { t } = useTranslation(); t('ns.key')`         |
| String in non-component code | `import i18n from '../i18n/config'; i18n.t('ns.key')` |
| Interpolate a value          | `t('ns.key', { name })` + `{{name}}` in JSON          |
| Pluralize                    | `key` + `key_plural`, `t('key', { count })`           |
| Add any new string           | Add to `en.json` AND `zh.json` at the same path       |
