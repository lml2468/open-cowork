# Skills, Commands, And Workflows

Skills and commands are textual entry points for user interaction with Coding.
Their core purpose is to tell the AI how to enter the Coding flow when the user
expresses a certain intent.

## Conceptual Differences

| Type    | Trigger mode                           | Best for                                                       |
| ------- | -------------------------------------- | -------------------------------------------------------------- |
| skill   | AI auto-match or explicit user mention | Long-term capabilities, workflow rules, modification guides.   |
| command | Explicit user invocation               | Clear operation entry points such as continue and finish-work. |

Coding workflow skills usually share one semantic set: brainstorm, before-dev,
check, update-spec, break-loop. Multi-file built-in skills such as `coding-meta`
use layered references.

## Common Paths

| Item     | Path                |
| -------- | ------------------- |
| Skills   | `.claude/skills/`   |
| Commands | `.claude/commands/` |

In a user project, use the files actually generated by init as authoritative.

## Skill Structure

A common skill is a directory:

```text
coding-meta/
├── SKILL.md
└── references/
```

`SKILL.md` should tell the AI:

- When to use this skill.
- Which reference to read first for the current task.
- What not to do.

References hold longer explanations so the entry file does not contain everything.

## Command Structure

Commands are usually single files. Their content should include:

- When to use it.
- Which `.coding/` files to read.
- Which scripts to run.
- How to report after completion.

They should not store task state; task state belongs in `.coding/tasks/` and
`.coding/.runtime/`.

## Local Change Scenarios

| User need                           | Edit location                                      |
| ----------------------------------- | -------------------------------------------------- |
| Change AI auto-trigger rules        | The corresponding skill's frontmatter description. |
| Change user command behavior        | The corresponding command file.                    |
| Add a project-local skill           | `.claude/skills/`.                                 |
| Change finish/continue entry points | `.claude/commands/`.                               |

## Modification Principles

1. **Keep entry files short; references carry long content**. This matters especially for multi-file skills like `coding-meta`.
2. **Make trigger descriptions specific**. A description that is too broad can mis-trigger; one that is too narrow may not trigger.
3. **Put project-specific capabilities in local skills**. Do not put team-private flows into public `coding-meta`.

If the user only wants local AI to know one more project rule, usually create a
project-local skill or update `.coding/spec/` instead of changing a Coding
built-in workflow skill.
